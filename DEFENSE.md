# Лабораторная работа 5. Бинарная куча — материалы к защите

---

## 1. Теория: АТД «Очередь с приоритетом»

**Очередь с приоритетом** (priority queue) — абстрактная структура данных, в которой каждый элемент имеет числовой приоритет (ключ). Первым извлекается элемент с наибольшим (max-heap) или наименьшим (min-heap) приоритетом.

### Поддерживаемые операции АТД

| Операция | Описание |
|---|---|
| **Insert** | Добавить элемент с заданным приоритетом |
| **Max / Min** | Вернуть элемент с максимальным/минимальным приоритетом без удаления |
| **ExtractMax / ExtractMin** | Удалить и вернуть элемент с максимальным/минимальным приоритетом |
| **IncreaseKey / DecreaseKey** | Изменить приоритет заданного элемента |
| **Merge** | Слить две очереди в одну |

### Варианты реализации

| Реализация | Insert | ExtractMax | IncreaseKey | Merge |
|---|---|---|---|---|
| Неупорядоченный массив | O(1) | O(n) | O(n) | O(n) |
| Упорядоченный массив | O(n) | O(1) | O(n) | O(n) |
| **Бинарная куча (массив)** | **O(log n)** | **O(log n)** | **O(log n)** | **O(n)** |
| Биномиальная куча | O(log n) | O(log n) | O(log n) | O(log n) |
| Фибоначчиева куча | O(1) aморт. | O(log n) aморт. | O(1) aморт. | O(1) |

Бинарная куча — оптимальный выбор для большинства практических задач: простая реализация, хорошие константы, O(n) построение из готового массива.

## 2. Теория: Бинарная куча

### Определение

**Бинарная куча** (binary heap, пирамида, сортирующее дерево) — это двоичное дерево, удовлетворяющее двум условиям:

1. **Свойство кучи (heap property):** приоритет любой вершины не меньше (≥) приоритета её потомков — для max-heap; не больше (≤) — для min-heap.
2. **Структурное свойство:** дерево является **полным бинарным деревом** (complete binary tree) — все уровни заполнены слева направо, возможно кроме последнего.

### Два вида куч

- **Max-heap:** `H[parent(i)] ≥ H[i]` — максимум в корне. Используется в алгоритме HeapSort, при извлечении элемента с наибольшим приоритетом.
- **Min-heap:** `H[parent(i)] ≤ H[i]` — минимум в корне. Используется в алгоритме Дейкстры, в данной лабораторной работе.

### Реализация на основе массива

Полное бинарное дерево компактно укладывается в массив без хранения указателей. Корень находится в ячейке с индексом **0**.

Для узла с индексом `i` (индексация с 0):

```
Parent(i)      = (i - 1) / 2    (целочисленное деление)
LeftChild(i)   = 2 * i + 1
RightChild(i)  = 2 * i + 2
```

Пример max-heap из 9 элементов `[99, 77, 88, 55, 57, 72, 61, 22, 33]`:

```
Дерево:           Массив:
      99           [0]=99  [1]=77  [2]=88
    /    \         [3]=55  [4]=57  [5]=72
  77      88       [6]=61  [7]=22  [8]=33
 /  \   /  \
55  57 72  61
/\
22 33
```

Проверка: H[Parent(4)] = H[1] = 77 ≥ H[4] = 57 ✓

**Структура данных в коде:**

```c
typedef struct {
    int   key;   // приоритет (ключ)
    char* val;   // значение (хранится по указателю, не копируется)
} heap_node;

typedef struct {
    int        cap;   // максимальная ёмкость (фиксирована при создании)
    int        len;   // текущее количество элементов
    heap_node* data;  // плоский массив узлов
} minheap;
```

## 3. Описание всех функций реализации

### 3.1 Вспомогательные функции навигации

#### `get_parent`, `get_l_child`, `get_r_child`

```c
int get_parent(int ind)   { return (ind - 1) / 2; }
int get_l_child(int ind)  { return 2 * ind + 1; }
int get_r_child(int ind)  { return 2 * ind + 2; }
```

Формулы прямо следуют из укладки полного бинарного дерева в массив с нулевым корнем. Целочисленное деление в `get_parent` гарантирует, что и левый потомок `(2i+1)`, и правый `(2i+2)` дают одного и того же родителя `i`.

**Сложность:** O(1) по времени, O(1) по памяти.

---

#### `heap_swap`

```c
void heap_swap(heap_node* a, heap_node* b) {
    heap_node temp = *a;
    *a = *b;
    *b = temp;
}
```

Копирует весь узел целиком (ключ + указатель на строку) через временную переменную. Так как `val` — это указатель, а не строка, копирование узла не копирует саму строку — это нормально.

**Сложность:** O(1).

---

### 3.2 Создание и освобождение кучи

#### `minheap_create(int cap)`

```c
minheap* minheap_create(int cap) {
    minheap* h = malloc(sizeof(minheap));
    if (h != NULL) {
        h->cap = cap;
        h->len = 0;
        h->data = malloc(sizeof(heap_node) * cap);
        if (h->data == NULL) { free(h); return NULL; }
    }
    return h;
}
```

Два вызова `malloc`: сначала под структуру-заголовок, затем под массив узлов. Если второй `malloc` падает, первый освобождается — утечки нет. Ёмкость фиксируется навсегда: куча не умеет расти.

**Сложность:** O(1) по времени, O(cap) по памяти.

---

#### `minheap_free(minheap* h)`

```c
void minheap_free(minheap* h) {
    if (h != NULL) {
        free(h->data);
        free(h);
    }
}
```

Порядок освобождения важен: сначала массив узлов `data`, затем сам заголовок. Обратный порядок создал бы висячий указатель. Значения `val` не освобождаются — они не принадлежат куче.

**Сложность:** O(1).

---

### 3.3 Внутренние процедуры просеивания

#### `heapify_up(minheap* h, int ind)` — просеивание вверх

```c
void heapify_up(minheap* h, int ind) {
    int p = get_parent(ind);
    while (ind > 0 && h->data[ind].key < h->data[p].key) {
        heap_swap(&h->data[ind], &h->data[p]);
        ind = p;
        p = get_parent(ind);
    }
}
```

**Идея:** новый элемент появился на позиции `ind`. Если он меньше своего родителя, он должен подняться выше. Повторяем пока не дойдём до корня или пока свойство min-heap не будет выполнено.

**Трассировка примера:** вставляем ключ 3 в кучу `[5, 10, 8, 15, 12]` (len=5):
- ind=5, p=2: data[5]=3 < data[2]=8 → swap → `[5, 10, 3, 15, 12, 8]`
- ind=2, p=0: data[2]=3 < data[0]=5 → swap → `[3, 10, 5, 15, 12, 8]`
- ind=0: корень, стоп.

**Сложность:** O(log n) — высота дерева не более ⌊log₂ n⌋.

---

#### `heapify_down(minheap* h, int ind)` — просеивание вниз

```c
void heapify_down(minheap* h, int ind) {
    while (1) {
        int left = get_l_child(ind), right = get_r_child(ind);
        int smallest = ind;
        if (left  < h->len && h->data[left].key  < h->data[smallest].key) smallest = left;
        if (right < h->len && h->data[right].key < h->data[smallest].key) smallest = right;
        if (smallest != ind) {
            heap_swap(&h->data[ind], &h->data[smallest]);
            ind = smallest;
        } else break;
    }
}
```

**Идея:** элемент на позиции `ind` нарушает свойство кучи снизу. Находим наименьшего из трёх: текущий, левый потомок, правый потомок. Меняем текущий с наименьшим и спускаемся в то поддерево.

**Трассировка extractmin:** куча `[1, 3, 2, 7, 5, 4]`. Извлекаем 1, кладём 4 в корень → `[4, 3, 2, 7, 5]`. heapify_down(0):
- ind=0: smallest(4,3,2)=2(индекс 2) → swap → `[2, 3, 4, 7, 5]`
- ind=2: left=5 (нет), right=6 (нет), smallest=2 → break.

Результат: `[2, 3, 4, 7, 5]` — корректная куча.

**Сложность:** O(log n).

### 3.4 Основные операции

#### `minheap_min(minheap* h)` — получить минимум

```c
heap_node* minheap_min(minheap* h) {
    if (h->len == 0) return NULL;
    return &(h->data[0]);
}
```

Минимум всегда в корне (индекс 0). Возвращается указатель на узел внутри массива — клиент не должен его хранить после изменения кучи.

**Сложность:** O(1).

---

#### `minheap_insert(minheap* heap, int key, char* val)` — вставка

```c
minheap* minheap_insert(minheap* heap, int key, char* val) {
    if (heap->len >= heap->cap) return heap; // переполнение — тихий выход
    heap->data[heap->len].key = key;
    heap->data[heap->len].val = val;
    heapify_up(heap, heap->len);             // heap->len — индекс нового элемента
    heap->len++;
    return heap;
}
```

**Алгоритм:**
1. Проверить наличие свободного места.
2. Записать новый узел в позицию `len` (следующую свободную ячейку).
3. Вызвать `heapify_up` от этой позиции.
4. Увеличить `len`.

Важно: `heapify_up` вызывается **до** инкремента `len`, потому что именно `len` является корректным индексом нового элемента в данный момент.

**Сложность:** O(log n) — определяется `heapify_up`.

---

#### `minheap_extractmin(minheap* heap)` — извлечь минимум

```c
minheap* minheap_extractmin(minheap* heap) {
    if (heap->len == 0) return heap;
    heap->data[0] = heap->data[heap->len - 1]; // последний → корень
    heap->len--;
    if (heap->len > 1) heapify_down(heap, 0);
    return heap;
}
```

**Алгоритм:**
1. Минимум — корень (индекс 0). Перезаписываем его последним элементом массива.
2. Уменьшаем `len` (последний элемент логически удалён).
3. Восстанавливаем свойство кучи через `heapify_down` от корня.

Почему именно последний? Потому что дерево должно оставаться полным бинарным. Удаление последнего листа не нарушает структуру.

**Сложность:** O(log n) — определяется `heapify_down`.

---

#### `minheap_decrease_key(minheap* heap, int key, int newkey)` — уменьшить ключ

```c
minheap* minheap_decrease_key(minheap* heap, int key, int newkey) {
    int ind = -1;
    for (int i = 0; i < heap->len; i++) {
        if (heap->data[i].key == key) { ind = i; break; }
    }
    if (ind == -1 || newkey >= key) return heap;
    heap->data[ind].key = newkey;
    heapify_up(heap, ind);
    return heap;
}
```

**Алгоритм:**
1. Линейный поиск узла по значению ключа (берётся первое вхождение).
2. Проверка: новый ключ должен быть строго меньше старого. Уменьшение ключа в min-heap может только поднять элемент выше.
3. Присваиваем новый ключ, вызываем `heapify_up`.

Почему только `heapify_up`, а не `heapify_down`? Поскольку ключ уменьшился, он стал «легче» и может нарушить свойство кучи только со стороны родителя (он стал меньше родителя), но не со стороны потомков (он был не меньше потомков, и стал ещё меньше).

**Сложность:** O(n) — линейный поиск + O(log n) просеивание = **O(n)**.

---

#### `minheap_delete(minheap* heap, int key)` — удалить элемент по ключу

```c
minheap* minheap_delete(minheap* heap, int key) {
    int ind = -1;
    for (int i = 0; i < heap->len; i++) {
        if (heap->data[i].key == key) { ind = i; break; }
    }
    if (ind == -1) return heap;
    heap->data[ind].key = INT_MIN; // −∞: узел всплывёт в корень
    heapify_up(heap, ind);
    minheap_extractmin(heap);      // удаляем корень
    return heap;
}
```

**Алгоритм — трюк через −∞:**
1. Найти узел линейным поиском.
2. Присвоить ключ `INT_MIN` (−∞) → он гарантированно меньше любого элемента.
3. `heapify_up` всплывает узел в корень.
4. `extractmin` удаляет корень (наш узел).

Это корректный способ удаления произвольного элемента в min-heap. Он использует уже реализованные операции и не требует отдельной логики.

**Сложность:** O(n) — линейный поиск доминирует.

---

#### `build_minheap(int* arr, int n)` — построить кучу за O(n)

```c
minheap* build_minheap(int* arr, int n) {
    minheap* h = minheap_create(n);
    if (h == NULL) return NULL;
    for (int i = 0; i < n; i++) { h->data[i].key = arr[i]; h->data[i].val = NULL; }
    h->len = n;
    for (int i = get_parent(n - 1); i >= 0; i--)
        heapify_down(h, i);
    return h;
}
```

**Алгоритм Флойда (Floyd's algorithm, bottom-up heap construction):**
1. Скопировать все n элементов массива в `data` как есть — это произвольный массив, не куча.
2. Листья (индексы > ⌊(n−1)/2⌋) — тривиальные кучи из одного элемента. Их не трогаем.
3. От последнего нелистового узла `⌊(n−1)/2⌋` до корня включительно вызвать `heapify_down`.

**Почему O(n), а не O(n log n)?**

Наивный подход: n раз вставить через `minheap_insert` → O(n log n).

Флойд: узлы на высоте `h` опускаются не более чем на `h` уровней. Число узлов на высоте `h` не превышает `⌈n / 2^(h+1)⌉`. Суммарная работа:

```
T(n) = Σ(h=0..⌊log n⌋) ⌈n/2^(h+1)⌉ · O(h)
     = O(n · Σ(h=0..∞) h/2^h)
     = O(n · 2)          [ряд h·x^h сходится к x/(1-x)² при x=1/2, x/(1-x)²=2]
     = O(n)
```

**Трассировка примера (n=7, массив [55,66,99,77,11,44,22]):**
- Последний нелистовой: i = Parent(6) = 2. data[2]=99, дети 44,22 → 22 < 99 → swap → 22 на позиции 2.
- i=1: data[1]=66, дети 77,11 → 11 < 66 → swap → 11 на позиции 1. Опускаем дальше: дети 77,66 → стоп.
- i=0: data[0]=55, дети 11,22 → 11 < 55 → swap. Продолжаем: бывший 55 на позиции 1, дети 77,66 → стоп.
- Результат: [11, 55, 22, 77, 66, 44, 99] — корректная min-heap.

**Сложность:** O(n) по времени, O(1) дополнительной памяти.

---

#### `minheap_union(minheap* heap1, minheap* heap2)` — слияние двух куч

```c
minheap* minheap_union(minheap* heap1, minheap* heap2) {
    int n = heap1->len + heap2->len;
    minheap* h = minheap_create(n);
    if (h == NULL) return NULL;
    for (int i = 0; i < heap1->len; i++) h->data[i] = heap1->data[i];
    for (int i = 0; i < heap2->len; i++) h->data[heap1->len + i] = heap2->data[i];
    h->len = n;
    for (int i = get_parent(n - 1); i >= 0; i--)
        heapify_down(h, i);
    return h;
}
```

**Алгоритм:**
1. Создать новую кучу ёмкостью `n = len1 + len2`.
2. Скопировать узлы обеих куч в новый массив (одна за другой).
3. Применить алгоритм Флойда к объединённому массиву.

Исходные кучи не изменяются и не освобождаются — управление ими остаётся за вызывающим кодом.

Обратите внимание: бинарная куча на массиве не умеет сливаться лучше чем O(n) — это её принципиальный недостаток. Биномиальная и фибоначчиева кучи делают Merge за O(log n) и O(1) соответственно.

**Сложность:** O(n) по времени, O(n) по памяти (новый массив).

## 4. Экспериментальное исследование

### Методология

Все замеры выполняются с `REPEAT = 10000` повторений, результат — среднее время одного запуска. Ключи — псевдослучайные числа в диапазоне [10 000, 1 000 000]. Время измеряется через `clock_gettime(CLOCK_MONOTONIC)` (монотонные часы, не чувствительны к корректировкам системного времени).

### Таблица 1: Добавление элементов

**Условие:** `minheap_insert` — n последовательных вставок в пустую кучу. `build_minheap` — однократный вызов для n элементов.

Ожидаемое соотношение: `build_minheap` быстрее `minheap_insert × n`, поскольку O(n) < O(n log n). При n = 400 000 разница примерно в log₂(400000) ≈ 18 раз.

### Таблица 2: Извлечение и уменьшение приоритета

**Условие (по заданию):** оба замера выполняются после того, как все n элементов добавлены в кучу. Для `minheap_decrease_key` дополнительно выполнено одно извлечение.

`minheap_extractmin` — O(log n), поэтому время растёт логарифмически. `minheap_decrease_key` — O(n) (линейный поиск), поэтому время растёт линейно: при удвоении n время удваивается.

## 6. Контрольные вопросы из задания с развёрнутыми ответами

### В1: Что такое очередь с приоритетом?

Очередь с приоритетом — это АТД, в котором каждый элемент имеет числовой приоритет (ключ), и операция извлечения всегда возвращает элемент с наибольшим (или наименьшим) приоритетом. В отличие от обычной очереди, порядок извлечения определяется не временем добавления, а ключом. Основные операции: Insert, Max/Min, ExtractMax/ExtractMin, IncreaseKey/DecreaseKey, Merge.

---

### В2: Варианты реализации очереди с приоритетом

1. **Неупорядоченный массив/список:** Insert — O(1), ExtractMax — O(n) (нужно найти максимум). Прост в реализации, плох для частых извлечений.
2. **Упорядоченный массив/список:** Insert — O(n) (нужно вставить на правильную позицию), ExtractMax — O(1). Плох для частых вставок.
3. **Бинарная куча на массиве:** Insert — O(log n), Extract — O(log n). Золотая середина, практически используется повсеместно.
4. **Биномиальная куча:** улучшает Merge до O(log n), все остальные операции тоже O(log n).
5. **Фибоначчиева куча:** Insert и DecreaseKey — O(1) амортизированно, ExtractMax — O(log n) амортизированно. Теоретически оптимальна для алгоритма Дейкстры, на практике сложна в реализации.

---

### В3: Объяснение механизма работы функций бинарной кучи

**heapify_up:** применяется после вставки нового элемента в конец массива. Новый элемент сравнивается с родителем; если он меньше (в min-heap) — меняются местами, процесс повторяется вверх. Останавливается у корня или когда свойство кучи восстановлено. За одну итерацию элемент поднимается на один уровень, уровней не более log n.

**heapify_down:** применяется после удаления корня (или при построении кучи Флойда). Элемент сравнивается с меньшим из двух потомков; если потомок меньше — меняются местами, спуск продолжается. Останавливается у листа или когда оба потомка ≥ текущего.

**build_minheap:** листья (правая половина массива) являются тривиальными кучами. Начиная с последнего нелистового узла и до корня применяем heapify_down. Корректность: после обработки узла i всё поддерево с корнем i является кучей (инвариант по i убывающему).

---

### В4: Вычислительная сложность операций бинарной кучи

Ключевой факт: высота полного бинарного дерева из n элементов равна **⌊log₂ n⌋**.

- **O(1):** получение минимума (корень), создание, освобождение.
- **O(log n):** вставка, извлечение минимума, heapify_up, heapify_down, decrease_key (если позиция узла известна).
- **O(n):** построение из массива (Флойд, лучше, чем O(n log n) наивного), слияние, decrease_key/delete с поиском по ключу.

Почему построение O(n), а не O(n log n)? Большинство узлов находится в нижних уровнях и «опускается» на малое число шагов. Точная сумма: Σ(h·n/2^h) ≤ 2n.

---

### В5: Операция слияния двух куч

В бинарной куче на массиве: скопировать оба массива подряд, применить алгоритм Флойда — итого O(n). Нельзя сделать лучше, не зная структуры элементов.

Это принципиальное ограничение бинарной кучи. Биномиальная куча делает Merge за O(log n) за счёт более сложной структуры (список биномиальных деревьев). Фибоначчиева куча — за O(1), просто сцепляя два списка корней без немедленного упорядочивания.

---

### В6: Как добавить элементы в бинарную кучу?

**Способ 1 — последовательная вставка:** для каждого элемента вызвать `minheap_insert`. Каждая вставка O(log n), n вставок — O(n log n).

**Способ 2 — алгоритм Флойда (build_minheap):** скопировать все n элементов в массив, затем применить heapify_down от последнего нелистового узла до корня. Сложность O(n). Это оптимальный способ, когда все элементы известны заранее.

В данной реализации: `build_minheap(arr, n)` — Флойд. Последовательная вставка через `minheap_insert` в цикле — наивный способ.

---

### В7: Область применения бинарной кучи

1. **Очередь с приоритетом** — базовое применение: планировщики задач ОС, системы реального времени.
2. **Алгоритм Дейкстры** — нахождение кратчайших путей в графе: на каждом шаге извлекается вершина с минимальным расстоянием.
3. **Алгоритм Прима** — нахождение минимального остовного дерева.
4. **HeapSort** — сортировка за O(n log n) с O(1) доп. памятью: построить max-heap, затем n раз extractmax.
5. **Алгоритм кодирования Хаффмана** — построение оптимального префиксного кода: сливать два поддерева с наименьшими весами.
6. **k-й наименьший элемент** — за O(n + k log n).
7. **Медиана в потоке** — два хипа (max и min).

## 7. Дополнительные теоретические вопросы (дотошный лектор)

### Т1: Почему нельзя за O(1) найти произвольный элемент в куче?

Куча гарантирует только порядок между родителем и потомками по вертикали. Между братьями (sibling nodes) порядок не определён. Например, в корректной min-heap можно иметь: `[1, 5, 2, 100, 7, 3, 8]`. Узел со значением 100 стоит перед узлами 3 и 8 в массиве. Нет никакого способа сказать, в какой половине массива находится нужный ключ — придётся проверять все. Отсюда O(n) для поиска.

---

### Т2: Докажите, что высота бинарной кучи из n элементов равна ⌊log₂ n⌋

Полное бинарное дерево высоты h содержит от 2^h (минимум — только корневой уровень заполнен, остальные пусты — нет, такого быть не может при полном дереве) до 2^(h+1)−1 узлов. Точнее: 2^h ≤ n ≤ 2^(h+1)−1. Отсюда: h ≤ log₂ n < h+1, то есть h = ⌊log₂ n⌋.

При n=1: h=0 (один корень, нет уровней ниже). При n=2: h=1. При n=7: h=2 (полное дерево). При n=8: h=3.

---

### Т3: Чем отличается `heapify_up` от `heapify_down` и когда какой применять?

| | `heapify_up` | `heapify_down` |
|---|---|---|
| Направление | снизу вверх, к корню | сверху вниз, к листьям |
| Когда применять | после вставки нового элемента, после decrease_key | после удаления корня, при построении кучи |
| Почему | новый/уменьшившийся элемент может быть меньше родителя | элемент, пришедший в корень, может быть больше потомков |
| Сравнение | с одним родителем | с двумя потомками, выбирается наименьший |

---

### Т4: Почему `heapify_down` сравнивает с меньшим из двух потомков, а не с любым?

Если в min-heap узел A нарушает свойство кучи (A > min(левый, правый)), нужно поменять его с потомком, у которого меньший ключ. Если поменять с большим потомком B, то на месте B окажется A > B, и свойство кучи уже нарушено между новым родителем B и его потомком A. Выбирая минимальный потомок, мы гарантируем, что после обмена новый родитель (бывший минимальный потомок) ≤ оставшемуся брату.

---

### Т5: Что происходит при попытке вставить элемент в заполненную кучу?

В данной реализации: функция проверяет `heap->len >= heap->cap` и возвращает `heap` без изменений (тихий выход). Это сознательное ограничение: ёмкость фиксируется при `minheap_create`. Элемент не вставляется, ошибка не сигнализируется. В реальных системах нужна либо динамически расширяемая куча (`realloc`), либо явный код ошибки.

---

### Т6: Почему при extractmin последний элемент перемещается в корень, а не первый?

Дерево должно оставаться **полным бинарным деревом** (structural property). Единственный узел, который можно «безболезненно» убрать без нарушения структуры — это последний лист (крайний правый на последнем уровне). Именно поэтому берётся `data[len-1]`. После перемещения в корень структурное свойство не нарушено (дерево с n-1 элементами по-прежнему полное), но может нарушиться свойство порядка — его восстанавливает `heapify_down`.

---

### Т7: Можно ли реализовать DecreaseKey за O(log n) вместо O(n)?

Да — если хранить дополнительный массив или хеш-таблицу `key → index`, отображающий ключ в его позицию в массиве кучи. Тогда поиск O(1), и суммарная сложность O(log n). Но при каждом `heap_swap` нужно обновлять этот индекс. Именно так работает куча с поддержкой decrease_key в реализациях алгоритма Дейкстры. В данной лабораторной работе эта оптимизация не реализована — поиск наивный, O(n).

---

### Т8: В чём принципиальное отличие build_minheap от n × insert?

`n × insert` строит кучу сверху вниз: каждый элемент добавляется в конец и поднимается вверх. Элементы глубоко в дереве проходят длинный путь наверх.

`build_minheap` (Флойд) строит снизу вверх: начинает с нелистовых узлов и опускает их вниз. Ключевое наблюдение: листьев половина, и они вообще не обрабатываются. Узлов на уровне h от листа всего n/2^(h+1), и каждый делает не более h обменов. Итоговая сумма — линейная.

`n × insert`: каждый из n элементов в худшем случае проходит path от листа до корня ≈ log n шагов → O(n log n).

---

### Т9: Чем val отличается от key? Могут ли два узла иметь одинаковый key?

`key` — числовой приоритет, по нему упорядочена куча. `val` — произвольная нагрузка (строка), которая «едет» вместе с узлом, не влияя на порядок.

Два узла могут иметь одинаковый ключ — куча это допускает (свойство ≤, а не строгое <). Но `minheap_decrease_key` и `minheap_delete` при дубликатах обрабатывают только первое вхождение (линейный поиск делает `break` на первом совпадении).

---

### Т10: Что такое «свойство кучи» и как его проверить для данного массива?

Для min-heap: для каждого узла i > 0 должно выполняться `data[parent(i)].key ≤ data[i].key`.

Алгоритм проверки: цикл `for (i = 1; i < len; i++) assert(data[(i-1)/2].key <= data[i].key)`. Сложность O(n).

Пример: массив `[1, 3, 2, 7, 5, 4, 6]`:
- i=1: parent=0, 1≤3 ✓
- i=2: parent=0, 1≤2 ✓
- i=3: parent=1, 3≤7 ✓
- i=4: parent=1, 3≤5 ✓
- i=5: parent=2, 2≤4 ✓
- i=6: parent=2, 2≤6 ✓ → корректная min-heap.

---

### Т11: Чем бинарная куча отличается от бинарного дерева поиска (BST)?

| | Бинарная куча | BST |
|---|---|---|
| Свойство порядка | между родителем и потомками | между левым, текущим, правым |
| Нахождение минимума | O(1) — корень | O(h) — самый левый лист |
| Поиск произвольного | O(n) | O(h), в среднем O(log n) |
| Извлечение минимума | O(log n) | O(h) |
| Структура | всегда полное бинарное дерево | может быть несбалансированным |
| Реализация | плоский массив | узлы с указателями |

BST хорош для поиска, куча — для priority queue.

---

### Т12: Почему функция называется `heapify`, а не `sort`?

«Heapify» означает «превратить в кучу» или «восстановить свойство кучи». Это именно то, что делает функция: она берёт поддерево, которое не является кучей (только корень нарушает свойство), и восстанавливает порядок, продвигая нарушитель вниз. Сортировка же упорядочивает все элементы — это другая операция с другой сложностью.
